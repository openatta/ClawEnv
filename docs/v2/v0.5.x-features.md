# v0.5.x Feature Backlog

Items deliberately deferred from v2-feature-complete. Each entry has a
clear owner (CLI vs GUI vs core) and a one-line rationale for why it
didn't make the cut.

## CLI

- **`clawcli logs --since <dur>`** — currently parsed but not honored.
  Implementation needs: per-line timestamp parsing for native logs,
  `journalctl --since` for sandboxed instances. Skipped for v2 because
  the host-side / Alpine-side log formats diverge enough that a clean
  time filter is its own design problem.

- **`clawcli upgrade --rollback`** — re-install the previous claw version
  if the upgrade probe fails. v2 just reports failure; rollback is a
  follow-up that requires keeping the previous package install
  artifact around.

- **`clawcli sandbox snapshot {create, list, restore, prune}`** — Lima
  has `limactl snapshot` underneath. `SandboxBackend::supports_snapshot`
  exists; only Lima returns true. Wire the four ops through and gate
  them on capability.

- **`clawcli native --uninstall-deps`** — bootstrap installs node/git
  under `~/.clawenv/{node,git}`; there's currently no clean way to
  remove them.

- **Disk-resize in-place for Lima primary disk** — today
  `sandbox edit --disk-gb` writes lima.yaml then asks the user to
  restart. Lima's `limactl disk resize` only operates on
  `additionalDisks`, so primary-disk resize requires the yaml-edit
  path. Could be automated end-to-end with a "stop → edit → start"
  composite verb.

- **`clawcli sandbox port edit`** — replace a forward atomically
  (currently `add` doesn't replace; the GUI works around it by
  removing then adding, which leaks for a brief window).

## GUI (Tauri + frontend)

- **Per-instance proxy override UI (wizard step)** — backend wiring
  landed in v0.4 (`InstanceConfig.proxy: Option<InstanceProxyConfig>`,
  `set_instance_proxy` IPC), but the install wizard's proxy step still
  only writes the *global* proxy. Need a "use a different proxy than
  the global one for this claw" toggle on the per-claw page.

- **OS-proxy auto-refresh** — `start_instance` re-reads OS system
  proxy on each launch (Native claws). Sandbox claws bake proxy
  into the VM at install time; if the user toggles Clash *between*
  install and launch, the sandboxed claw won't see the new proxy
  until reinstall. Real fix is a per-launch proxy injection step.

- **Bridge admin UI** — AttaRun bridge has an HTTP admin API
  (`POST /api/exec/approve`, `/api/hil/complete`); Tauri only wires
  HIL today. The full bridge admin (rule edit, allow/deny lists,
  permissions) is still gated behind the bridge daemon's own UI.

- **Live `clawcli logs --follow` panel** — clawcli supports streaming
  logs as of v2; the Tauri log panel still uses one-shot
  `get_instance_logs`. Wire to `cli_bridge::run_cli_streaming` and
  forward each Data event line to the panel.

- **Sandbox snapshot UI** — depends on the CLI-side snapshot verbs
  above.

- **Doctor auto-fix UI** — `clawcli doctor --fix` exists, but the
  GUI's doctor page doesn't yet have a "Fix all" button.

- **Per-claw upgrade rollback** — see CLI item.

## Core / wire

- **`StatusResponse.gateway_token`** — dropped from the v2 wire shape
  because it leaked into shell history when callers used
  `clawcli status --json | jq`. Tokens now flow only through
  `clawcli token`. The GUI's `getInstanceToken()` uses `clawcli token`
  directly, so this is informational.

- **`SystemInfo.checks` schema** — `info_only` is a v1-style hint for
  "this isn't actually a problem". v2 should split into severity:
  `info`/`warn`/`error` so the UI can render three colors instead of
  the current binary ok/red.

- **Idle timeout for streaming verbs** — CLI-DESIGN §4.4 specifies a
  client-side idle-timeout convention; clawcli doesn't enforce it
  from the server side yet (a hung VM exec can stall a JSON consumer
  indefinitely).

## Migration / cleanup

- ~~**v1 root workspace removal**~~ — done in v0.4: `legacy/core/`
  removed, Tauri ported to `clawops_core` end-to-end, workspace
  members are now `core` / `cli` / `tauri` only. Browser HIL HTTP
  server (~1100 LoC) lifted into `tauri/src/bridge_server/` as a
  GUI-private module pending the AttaRun bridge daemon takeover.

- **`clawenv-bundle.toml` schema v3** — current bundle manifest is the
  v1 schema reused. v2 adds new fields (claw entrypoint, MCP descriptor)
  that should ideally be a versioned bump.

- **`InstanceConfig.browser.vnc_ws_port` allocation** — v2 install
  pipeline doesn't allocate a vnc_ws port at install time even when
  `--browser` is passed; field is set to 0 and HIL noVNC depends on
  it. Either allocate at install (matches v1 behaviour) or compute
  deterministically from gateway_port (e.g. +5000).
